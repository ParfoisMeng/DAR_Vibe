## 1. 开发总览

```
Phase 0   │ 2周  │ 核心假设验证（Vanilla TS 原型）
Phase 1   │ 8周  │ 垂直切片（完整Run体验，Vue 3迁移）
Phase 2   │ 6月  │ EA（全内容扩展，云存档，社区功能）
Phase 3   │ 1年+ │ 正式版（全内容，移动端优化）
```

**核心原则**：
- 每个 Phase 结束都必须是**可玩、可演示**的版本
- 不提前开发后期功能，Phase 0/1 的所有代码允许一次性重写
- Phase 0 的 Go/No-Go 决策门通过前，不启动 Phase 1

## 2. Phase 0：核心循环原型（第1~14天）

### 目标

证伪或证实 4 个核心假设（H1~H4），决定是否进入 Phase 1。

### 技术约束

- 纯 HTML + Vanilla TypeScript，零依赖（可用 esbuild 转译）
- 单文件结构可接受（`index.html` + `game.ts`），快速迭代优先
- 数据写在 TypeScript 常量里，不做 JSON 分离

### Week 1：战斗引擎 + 核心流程

---

#### Day 1（战斗结算引擎）

**目标**：实现战斗自动结算的核心算法，能在控制台输出结算结果。

| 任务 | 验收标准 |
|-----|---------|
| 实现 `calcCombat(player, enemy)` 函数 | 输入双方属性，输出胜负、耗时、日志数组 |
| 实现 DPS 公式：`DPS = ATK × Speed × CritCoeff × SkillCoeff` | 数值与 GDD 10.1 基础数值吻合（筑基期 DPS ≈ 500/s） |
| 实现日志生成：每步攻击生成 1 条日志文本 | 日志使用仙侠动词模板（至少 10 条模板，含变量占位符） |
| 筑基期普通怪（HP 3000~6000）在 6~15 秒内结算完毕 | 跑 20 场战斗，所有结算时间在范围内 |

#### Day 14（内部游玩测试）

**目标**：通过 5~10 人游玩，评估 H1~H4 四个假设。

| 测试项 | 观测方法 | 通过标准 |
|-------|---------|---------|
| H1：3选1决策感 | 观察每次选择时的犹豫行为 | 80% 选择前玩家有停顿思考 |

| 重开欲望 | 死亡后直接问「还想继续玩吗」| 80% 以上回答「想」|

**Go/No-Go 决策**：
- 4 个假设全部通过 → 进入 Phase 1
- 3 个通过 1 个失败 → 分析失败假设，小范围修改后重测（最多 3 天）
- ≥2 个假设失败 → 暂停，重新审视核心设计

## 附录：协同日志语言自查清单

> 协同触发日志必须通过以下检查，否则需要修改（Q2 规范要求）。

**禁止出现的词汇**：

- 机制类：「激活」「触发」「效果」「百分比」「加成」「系数」「协同」
- 数值类：「+X%」「×X倍」「额外X点」（数值可在独立的「效果数字」区域显示）
- 系统类：「已解锁」「已激活」「检测到」

**必须包含的元素**：

- ≥1 个仙侠动词（「流转」「相生」「化合」「归元」「贯通」）
- ≥1 个仙侠意象（「天机」「道」「因果」「境」「气」「意」）
- 描述的是「发生了什么」而非「触发了什么」

**示例对照**：

| ❌ | ✅ |
|---|---|
| 「协同效果已激活，剑意叠加+100%」 | 「剑意与虚空相合，道意大盛，积累倍增」 |
| 「检测到双节点组合，效果生效」 | 「天机流转·两仪相生：阴阳既济，威力倍出」 |
| 「史诗协同触发成功」 | 「【天命感应】道韵震颤，此局已入化境」 |
